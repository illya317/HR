# 车间报表 Excel 转 JSON 校验整体计划

## 1. 目标

为车间报表建立一套可重复执行的验收流程，确保其他 agent 从 Excel 转出的 JSON 没有明显遗漏、错配、错算或不可解释的清洗问题。

最终目标不是只证明 JSON 格式正确，而是证明：

- Excel 原始报表中的关键记录都能在 JSON 中找到。
- 财务报表、工分明细、关联结果之间的数量和金额口径可对账。
- 异常记录被明确列出，并能区分转换错误与业务例外。
- 后续成本分摊分析使用的是可追溯、可复核的数据。

## 2. 输入文件

当前以 2025 年车间报表为样板，后续可扩展到 2026 年及其他年度。

| 文件 | 角色 | 说明 |
|---|---|---|
| `2025年车间报表.xlsx` | 源文件 | 原始 Excel，包含财务报表和 1-12 月工分明细 |
| `2025年车间报表_财务报表.json` | 中间产物 | 财务报表 JSON |
| `2025年车间报表_工分明细.json` | 中间产物 | 工分明细 JSON |
| `2025年车间报表_关联.json` | 主验收产物 | 财务批次挂载人员工分明细后的结果 |

## 3. 输出文件

计划实现一个校验脚本，固定输出两类报告：

| 文件 | 用途 |
|---|---|
| `2025年车间报表_校验报告.json` | 机器可读，供后续脚本或 agent 判断是否通过 |
| `2025年车间报表_校验报告.md` | 人可读，列出错误、警告、业务例外和汇总统计 |

可选输出：

| 文件 | 用途 |
|---|---|
| `2025年车间报表_校验明细.csv` | 适合导入 Excel/飞书表格筛查问题 |
| `2025年车间报表_异常批次.csv` | 只保留需要人工确认的批次 |

## 4. 验收原则

### 4.1 不直接信任转换结果

校验脚本应从 Excel 原表独立抽取关键事实，再与 JSON 对账。不能只在 JSON 内部自洽检查，否则发现不了转换 agent 漏读、错读、误合并的问题。

### 4.2 区分错误和业务例外

例如：

- 空批号可能是 Excel 中合并批次或补充行，不一定是错误。
- 一个工分明细挂到多个财务批次，可能是多批共用一张工分单。
- 财务工分和人员合计不一致，可能是跨月归属，也可能是错挂。

脚本应把这些列为 `WARN` 或 `REVIEW`，而不是全部直接判死刑。

### 4.3 以可追溯为底线

每条异常最好能给出：

- 月份
- 品种
- 批号
- 财务工分
- 人员工分合计
- 差异
- 来源 sheet
- 来源行号或工分明细摘要
- 异常类型

## 5. 校验范围

### 5.1 文件级校验

- Excel 文件是否存在且可打开。
- JSON 文件是否存在且可解析。
- JSON 顶层结构是否符合预期。
- 文件年份、月份、sheet 名称是否一致。
- Excel 是否包含：
  - `2025年财务报表`
  - `1月工分明细` 到 `12月工分明细`

### 5.2 财务报表 JSON 校验

检查对象：`2025年车间报表_财务报表.json`

校验项：

- 是否包含 1-12 月。
- 每月是否有 `报表标题`、`品种记录`。
- 每个品种是否包含 `序号`、`品种`、`规格`、`批次记录`。
- 每个批次是否包含：
  - `批号`
  - `投料量_万粒`
  - `工分`
  - `生产数量`
  - `折合_盒瓶`
  - `折合_万粒`
  - `备注`
- 数字字段是否为数字或明确为空。
- 每月 JSON 批次数是否与 Excel 财务报表独立抽取的批次数一致。
- 每月 JSON 工分合计是否与 Excel 财务报表合计行一致。

### 5.3 工分明细 JSON 校验

检查对象：`2025年车间报表_工分明细.json`

校验项：

- 是否包含 1-12 月。
- 每月工分明细条目数是否与 Excel 中识别出的工分块数量一致。
- 每个工分块是否包含：
  - `品名`
  - `批号信息`
  - `人员工分明细`
- 每个人员明细是否包含：
  - `岗位类别`
  - `姓名`
  - `每日工分`
  - `合计`
- `每日工分` 求和是否等于 `合计`。
- 工分块人员 `合计` 总和是否等于 Excel 工分块底部合计。
- 岗位类别是否存在异常空值；如果为空，是否可由上一行继承。
- 姓名是否为空。
- 每日工分是否包含非数字、文本残留或错列。

### 5.4 关联 JSON 校验

检查对象：`2025年车间报表_关联.json`

校验项：

- 是否包含 1-12 月。
- 总批次数是否与财务报表 JSON 一致。
- 每个财务批次是否最多出现一次。
- 每个工分明细是否被关联或进入 `未匹配工分明细`。
- `未匹配工分明细` 是否为空；若不为空，列为 `ERROR` 或 `REVIEW`。
- 已关联工分明细数量是否等于工分明细 JSON 总条目数。
- 已匹配人员人次是否等于工分明细 JSON 人员总数。
- 财务批次挂载的 `工分明细` 是否存在重复挂载。
- 同一工分明细挂载到多个财务批次时，必须标记为 `multi_batch_detail`。

## 6. 批号解析规则

批号是本项目最容易出错的字段，需要单独固化规则。

### 6.1 单批号

示例：

- `20250101`
- `批号：20250101( 投料量150万粒）`

应解析为：

- `20250101`

### 6.2 同月范围批号

示例：

- `20250111-12`
- `20250502-0507`

应解析为：

- `20250111`, `20250112`
- `20250502`, `20250503`, `20250504`, `20250505`, `20250506`, `20250507`

### 6.3 斜杠组合

示例：

- `20250212/20250224-26`

应解析为：

- `20250212`, `20250224`, `20250225`, `20250226`

### 6.4 跨段组合

示例：

- `20251201-1205/18`

需要明确规则。建议解析为：

- `20251201`, `20251202`, `20251203`, `20251204`, `20251205`, `20251218`

如果转换 agent 使用了不同规则，校验报告必须列出差异。

### 6.5 空批号

财务报表中可能出现空批号行。处理规则：

- 若该行有工分、生产数量或备注，则保留为空批号批次，并标记 `empty_batch_no`。
- 若该行是合计、制表人、复核人等非数据行，不应进入 JSON 批次。

## 7. 品种名称归一化规则

品种名称匹配需要保留原始名称，同时提供归一化名称。

建议维护一个显式映射表，而不是只靠模糊匹配。

当前已知归一化：

| 原始写法 | 归一化 |
|---|---|
| 盐酸氧氟沙星胶囊 | 盐酸左氧氟沙星胶囊，或按业务确认后的标准名 |
| 盐酸左氧氟沙星胶囊 | 盐酸左氧氟沙星胶囊 |
| 甘草酸二胺胶囊 | 甘草酸二铵胶囊 |
| 甘草酸二铵胶囊 | 甘草酸二铵胶囊 |

注意：名称归一化不能覆盖原始字段。报告中应同时显示 `原始品种名` 和 `归一化品种名`。

## 8. 异常分级

### 8.1 ERROR

满足以下条件之一，应判定为错误，默认不允许进入成本分摊：

- JSON 无法解析。
- Excel 必要 sheet 缺失。
- 月份缺失。
- 财务批次数和 Excel 抽取结果不一致。
- 工分明细条目数和 Excel 抽取结果不一致。
- 人员明细缺失姓名。
- 人员 `合计` 不是数字，且 Excel 中对应单元格是数字。
- 工分明细未匹配，且无法被业务例外规则解释。
- 同一财务批次重复出现且无合理解释。

### 8.2 WARN

满足以下条件之一，应列为警告，需要复核但不一定阻断：

- 财务批次空批号。
- 财务批次空工分。
- 一个工分明细挂到多个财务批次。
- 财务工分和人员合计不一致。
- 人员 `每日工分` 求和不等于 `合计`。
- 品种名称经过归一化才匹配。
- 批号经过范围解析才匹配。
- 跨月批次归属导致月度差异。

### 8.3 INFO

满足以下条件之一，仅作记录：

- 每月批次数。
- 每月工分明细数。
- 每月人员人次。
- 每月财务工分合计。
- 每月人员工分合计。
- 已关联批次数。
- 未关联批次数。

## 10. 校验脚本设计

建议脚本名：

```bash
validate_workshop_report.py
```

建议位置：

```text
/Users/koito/Desktop/.财务数据库/成本分析飞书实验版/validate_workshop_report.py
```

建议命令：

```bash
python validate_workshop_report.py \
  --excel 2025年车间报表.xlsx \
  --finance-json 2025年车间报表_财务报表.json \
  --work-json 2025年车间报表_工分明细.json \
  --linked-json 2025年车间报表_关联.json \
  --out-prefix 2025年车间报表
```

### 10.1 脚本模块

| 模块 | 职责 |
|---|---|
| 参数解析 | 接收 Excel、JSON、输出路径 |
| Excel 读取 | 用 `openpyxl` 读取源表 |
| 财务表抽取 | 从 Excel 财务报表 sheet 抽取月份、品种、批次、合计 |
| 工分明细抽取 | 从每月工分明细 sheet 抽取工分块、人员、合计 |
| JSON 加载 | 加载并标准化三个 JSON |
| 批号解析 | 解析单批号、范围批号、斜杠组合 |
| 品种归一化 | 使用显式映射表归一化品种名 |
| 对账引擎 | 执行文件级、月度级、批次级、人员级校验 |
| 异常收集 | 统一记录 `severity/type/message/context` |
| 报告输出 | 生成 JSON、Markdown、可选 CSV |

### 10.2 issue 结构

建议每个异常统一为：

```json
{
  "severity": "WARN",
  "type": "batch_work_points_mismatch",
  "message": "财务工分与挂载人员合计不一致",
  "month": 1,
  "product": "泮托拉唑钠肠溶片",
  "batch_no": "20250111",
  "finance_points": 682.5,
  "people_points": 1469,
  "diff": 786.5,
  "source": {
    "sheet": "1月工分明细",
    "row_start": null,
    "row_end": null
  }
}
```

### 10.3 summary 结构

建议报告顶层包含：

```json
{
  "status": "REVIEW",
  "generated_at": "2026-05-28T00:00:00+10:00",
  "inputs": {},
  "summary": {},
  "month_summary": [],
  "issues": []
}
```

状态规则：

- `PASS`：无 `ERROR`，无必须人工确认的 `WARN`。
- `REVIEW`：无 `ERROR`，但有 `WARN` 或业务例外。
- `FAIL`：存在 `ERROR`。

## 11. 成本分摊前置判断

在使用关联 JSON 做成本分摊前，建议必须满足：

- `status` 不能是 `FAIL`。
- 所有 `ERROR` 为 0。
- 所有 `batch_work_points_mismatch` 已确认处理方式。
- 所有 `multi_batch_detail` 已有分摊规则。
- 跨月批次差异已确认归属月份。
- 人员每日工分与合计不一致的记录已确认以哪个字段为准。

建议的业务规则：

| 场景 | 建议处理 |
|---|---|
| 单个工分明细对应单个财务批次 | 可直接按人员合计分摊 |
| 单个工分明细对应多个财务批次 | 先在批次间分摊工分，再分摊到人员 |
| 财务工分与人员合计不一致 | 默认以财务工分为总额，人员合计作为权重 |
| 人员每日工分求和不等于合计 | 默认以 `合计` 为准，并保留异常 |
| 跨月批次 | 按财务报表月份归属，工分明细来源月份单独记录 |

## 12. 实施阶段

### Phase 1：固化验收口径

- 确认批号范围解析规则。
- 确认品种名称归一化表。
- 确认每日工分与合计不一致时以哪个字段为准。
- 确认多批共用工分明细时的成本分摊规则。

交付物：

- 本计划文档更新版。
- `validation_rules.json` 或脚本内常量。

### Phase 2：实现基础校验脚本

- 加载 Excel 和 JSON。
- 生成文件级、月份级、记录数级校验。
- 输出 Markdown 和 JSON 报告。

交付物：

- `validate_workshop_report.py`
- `2025年车间报表_校验报告.json`
- `2025年车间报表_校验报告.md`

### Phase 3：实现 Excel 独立抽取

- 从财务报表 sheet 独立抽取批次数据。
- 从工分明细 sheet 独立抽取工分块和人员数据。
- 与 JSON 转换结果进行对账。

交付物：

- 抽取结果摘要。
- Excel vs JSON 差异清单。

### Phase 4：实现关联质量校验

- 校验财务批次与工分明细的挂载关系。
- 标记多批共用工分明细。
- 标记跨月批次。
- 标记财务工分与人员合计差异。

交付物：

- 关联质量报告。
- 异常批次清单。

### Phase 5：形成验收流程

- 写入固定执行命令。
- 明确 `PASS / REVIEW / FAIL` 规则。
- 给其他 agent 规定转换后必须跑校验。

交付物：

- 最终版校验报告。
- 后续年度复用说明。

## 13. 通过标准

一份转换结果可进入成本分摊分析，至少应满足：

- JSON 文件均可解析。
- 1-12 月均存在。
- 财务批次数与 Excel 抽取一致。
- 工分明细条目数与 Excel 抽取一致。
- 工分明细未匹配数量为 0，或未匹配项全部被确认排除。
- 人员明细没有姓名缺失。
- 所有 `ERROR` 为 0。
- 所有 `WARN` 已在报告中列出并有处理建议。
- 报告状态为 `PASS` 或经人工确认的 `REVIEW`。
